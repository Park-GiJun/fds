## 3. Performance Review (이도윤)

### 성능 이슈
| # | 심각도 | 파일 | 설명 |
|---|---|---|---|
| 1 | CRITICAL | `RateLimitFilter.kt:16` | ConcurrentHashMap 무한 증가 미수정 — 100만 IP → ~100MB+ 힙, OOM 위험. Caffeine Cache 즉시 교체 필요 |
| 2 | HIGH | `GeneratorService.kt:54-64` | `start()`에 Semaphore 미적용 — rate=1000이면 초당 1000 코루틴 무제한 생성. IO 스레드 풀 포화 위험 |
| 3 | MEDIUM | `RateLimitFilter.kt:40` | `@Synchronized`는 OS 모니터 락 — 동일 IP burst 시 컨텍스트 스위칭 비용. CAS 기반 lock-free 권장 |
| 4 | LOW | `KtorTransactionSendAdapter.kt:35` | `toRequest()` 건당 1개 추가 객체 할당 — 10K TPS에서 Minor GC 주기 단축 가능 |

### 최적화 제안
1. ConcurrentHashMap → Caffeine Cache (`maximumSize=100K`, `expireAfterAccess=10m`)
2. `start()` 코루틴에도 Semaphore 또는 `limitedParallelism(N)` 적용
3. 향후 TokenBucket CAS 전환 검토

### 심각도: HIGH
### 총 이슈 수: 4건 (Critical 1, High 1, Medium 1, Low 1)

## Quality Lead 종합 (한소율)

### 품질 판정
코드 품질 자체는 확실히 개선되었습니다. Spring 비의존 패턴, Semaphore bounded burst, camelCase 통일 등 좋은 선례가 만들어졌습니다. 하지만 **테스트 0%가 지속**되는 것이 가장 큰 문제입니다.

### 기술 부채 등록 항목
- CommonApiResponse/DomainExceptions 위치 재검토 (fds-common → 별도 모듈 또는 인프라 계층)
- Port 반환 타입 application DTO화 (GeneratorStatus → GeneratorStatusResult)
- burst() 응답 타입 통일 (Map<String, Any> → BurstResponse)

### 학습 포인트
1. **헥사고날 @Bean 등록 패턴**: application 계층 Spring 비의존을 위한 infrastructure config 활용법
2. **공통 모듈 책임 원칙**: fds-common은 Kafka 이벤트 스키마 전용. HTTP 응답 래퍼는 인프라 계층에 배치

### 품질 심각도: HIGH
