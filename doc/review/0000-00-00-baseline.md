## 3. Performance Baseline

### 인프라 설정 현황

| 컴포넌트 | 현재 설정 | 문제 |
|---|---|---|
| Kafka | 단일 브로커, 파티션 1(기본값), consumer concurrency 미설정 | 10K TPS 불가 |
| Elasticsearch | 512MB 힙, 단일 노드 | GC 스래싱 → OOM 예상 |
| Redis | maxmemory 미설정, noeviction 기본 정책 | 메모리 가득 시 쓰기 거부 |
| PostgreSQL | 기본 설정, HikariCP maxPoolSize=10(기본) | 이론적 최대 1,000 TPS |
| Docker | 컨테이너 리소스 제한 없음 | OOM Killer 위험 |

|---|---|---|
| PERF-001 | CRITICAL | RateLimitFilter ConcurrentHashMap 무한 증가 — OOM 위험 |

| PERF-003 | CRITICAL | Kafka 단일 파티션 → 컨슈머 1스레드 처리 한계 |

1. `spring.datasource.hikari.maximum-pool-size=50`
2. Kafka 토픽 파티션 12개 이상 + `spring.kafka.listener.concurrency=12`
3. ES 힙 최소 4GB (`ES_JAVA_OPTS=-Xms4g -Xmx4g`)
4. RateLimitFilter에 TTL 기반 정리 로직 추가
5. Generator burst에 Semaphore/chunked 방식 동시성 제한

## Quality Lead 종합

### 품질 판정: **기초 미달** (테스트 0%, 코드 컨벤션 일부 미확정)

구현된 코드(특히 fds-generator)의 코드 품질은 양호하나, 테스트 전무 + 컨벤션 미확정 + 도메인 규칙 불일치로 품질 기반이 부재하다.

### 확정된 코드 컨벤션

| 항목 | 기준 |
|---|---|
| 패키지 구조 | `com.gijun.fds.{module}.{layer}.{sublayer}` |
| 웹 어댑터 | `{Name}WebAdapter` |
| UseCase 구현체 | `{Resource}Service` |
| in-port | `{Resource}UseCase` |
| out-port | `{Action}{Resource}Port` |
| 상수 | `const val` → SCREAMING_SNAKE_CASE / 일반 `val` → camelCase |
| 함수 스타일 | 단일 표현식 → expression body, named argument + trailing comma 필수 |
| Logger | `LoggerFactory.getLogger(javaClass)` (향후 kotlin-logging 전환 검토) |

### 테스트 전략 확정

1. 순수 도메인 로직: JUnit 5 단위 테스트 (외부 의존 없음)
2. 코루틴 로직: kotlinx-coroutines-test + MockK
3. HTTP 어댑터: ktor-client-mock / WireMock
4. DB/Kafka/Redis 통합: Testcontainers
5. 탐지 규칙: 구현 시 TDD 선행 필수

## 확정된 컨벤션

- 패키지: `com.gijun.fds.{module}.{layer}.{sublayer}`
- 웹 어댑터: `{Name}WebAdapter`
- UseCase 구현체: `{Resource}Service` (Handler 아님)
- in-port 인터페이스: `{Resource}UseCase`
- out-port 인터페이스: `{Action}{Resource}Port`
- 상수: `const val` → SCREAMING_SNAKE_CASE / 일반 `val` → camelCase
- 단일 표현식 함수: expression body 사용
- 생성자/함수 호출: named argument + trailing comma 필수
- Logger: `LoggerFactory.getLogger(javaClass)` 인스턴스 패턴
- 커밋 메시지: 한국어

## 도메인 용어 사전

| 한국어 | 영어 (코드) | 설명 |
|--------|------------|------|
| 거래 | Transaction | 카드/결제 단건 거래 이벤트 |
| 이상거래 | FraudTransaction | 사기 또는 비정상으로 판정된 거래 |
| 탐지 | Detection | 거래의 이상 여부를 규칙 엔진으로 판정하는 행위 |
| 탐지 결과 | DetectionResult | 규칙 평가 후 생성되는 리스크 레벨 + 스코어 집합 |
| 위험도 | RiskLevel | LOW / MEDIUM / HIGH / CRITICAL 4단계 |
| 리스크 스코어 | RiskScore | 0~100, 트리거된 규칙들의 기여 점수 합산 |
| 유저 행동 프로필 | UserBehaviorProfile | Redis에 저장된 유저의 최근 거래 패턴 요약 |
| 탐지 규칙 | DetectionRule | 단일 이상 패턴을 평가하는 함수형 인터페이스 |
| 규칙 결과 | RuleResult | 단일 규칙의 트리거 여부 + 기여 점수 + 사유 |
| 알림 | Alert | 탐지 결과(HIGH/CRITICAL)에 의해 생성된 처리 항목 |
| 알림 상태 | AlertStatus | OPEN / REVIEWING / CONFIRMED / DISMISSED |
| 거래 상태 | TransactionStatus | PENDING / APPROVED / SUSPICIOUS / BLOCKED |
| 고액 거래 규칙 | HighAmountRule | 평균 거래금액의 5배 초과 시 스코어 40 부여 |
| 연속 거래 규칙 | RapidSuccessionRule | 5분 내 5건 이상 시 스코어 30 부여 |
| 불가능 이동 규칙 | GeoImpossibleTravelRule | 직전 거래와 900km/h 초과 이동 속도 시 스코어 50 부여 |
| 새벽 고액 규칙 | MidnightTransactionRule | 00~05시 + 50만원 이상 시 스코어 25 부여 |
| 사기 유형 | FraudType | 제너레이터가 생성하는 이상 거래 시나리오 분류 |
| 가맹점 | Merchant | 카드 결제가 발생한 사업자 |
| 가맹점 카테고리 | MerchantCategory | CAFE, GROCERY, ONLINE 등 업종 분류 |
| 카드번호 | CardNumber (PAN) | 16자리 결제 카드 식별 번호. Kafka 전송 시 마스킹 필요 |
| 슬라이딩 윈도우 | Sliding Window | 연속 시간 구간 내 이벤트 집계 방식 |
| 중복 알림 방지 | Alert Deduplication | Redis를 통해 동일 거래/유저에 대한 알림 중복 생성 방지 |
| 제너레이터 | Generator | 테스트용 거래 데이터를 생성/전송하는 독립 서비스 |
| 거래 이벤트 | TransactionEvent | Kafka `transaction-events` 토픽에 발행되는 거래 발생 메시지 |
| 탐지 결과 이벤트 | DetectionResultEvent | Kafka `detection-results` 토픽에 발행되는 탐지 완료 메시지 |
